# 調剤薬局LINEフォローアップサービス 要件定義書

**作成日**: 2025年3月3日  
**対象**: 薬局オーナー向け（非エンジニア）  
**前提**: LINE公式アカウントはすでにあり、Webアプリ開発は経験あり

---

## 1. 結論（このシステムでできること）

| できること | 一言で |
|-----------|--------|
| 処方箋の流れを一元管理 | 受付→準備中→完了をWeb画面で把握 |
| 患者への自動通知 | 調剤完了時にLINEで「お薬のご用意ができました」 |
| フォローアップの自動化 | 抗生物質など渡した薬に応じて、3日後など指定日にLINEで体調確認 |
| 患者の返信を薬剤師へ通知 | 患者が「体調良い/悪い」など返信→薬剤師に届く |
| 患者とのチャット | LINE上で薬剤師と患者がやり取り |
| 一斉送信 | 休業案内など、友だち全員や指定患者へ一斉にメッセージ送信 |

**患者登録**: なるべく簡単に（名前のみ必須、処方箋登録時に即登録可能）。  
**フォローアップ文例**: パターン数は増やせる設計。文例・選択肢はフリー入力で自分で作成し、いつでも編集可能。

**患者からのメッセージ**: 選択式をメインにし、手入力も可能。体調確認などはボタンタップで、詳細は自由入力で対応。

**補足の重要ルール**（運用前提）:
- 処方箋の原本は来局時に必要（事前送信は準備用途のみ）
- 患者のLINE紐付けはQRコードで行う
- 既存のLINE公式アカウントをそのまま使う

---

## 2. 全体像（何がどこで動くか）

```
┌─────────────────────────────────────────────────────────────────┐
│                    薬剤師（あなた）                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Web管理画面（PC・タブレット）                              │   │
│  │  ・処方箋一覧・ステータス変更・フォロー登録・チャット確認    │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 操作・データ保存
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  Webアプリ（Next.js + Supabase + Vercel）                         │
│  ・処方箋DB・患者DB・フォロー予定・LINE連携API                    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ LINE Messaging API
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  LINE公式アカウント（既存）                                        │
│  ・患者へメッセージ送信・患者からの返信受信・Webhook               │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ 友だち追加・メッセージ
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  患者                                                             │
│  ・LINEアプリで通知受信・体調返信・薬剤師とチャット               │
└─────────────────────────────────────────────────────────────────┘
```

**用語の解説**（初めての方向け）:
- **Webhook**: LINEが「患者がメッセージを送った」などのイベントを、あなたのサーバーに自動で通知する仕組み
- **Messaging API**: LINE公式アカウントと外部システム（このWebアプリ）を連携させるための仕組み
- **プッシュメッセージ**: 薬局側から患者へ「先に」送るメッセージ（料金対象）
- **応答メッセージ**: 患者が先に送ったメッセージへの返信（無料）

## 3. 機能要件（詳細）

### 3.1 薬剤師側（管理画面）

#### 3.1.1 処方箋の受信一覧

| 項目 | 内容 |
|------|------|
| 目的 | 来局前・来局後の処方箋を一覧で把握する |
| 表示項目 | 患者名、処方箋受付日時、ステータス、処方元医療機関、薬名（任意） |
| 並び順 | 受付日時の新しい順（デフォルト） |
| 検索・フィルタ | 患者名・日付・ステータスで絞り込み |
| 入力方法 | 手動登録（処方箋受付時に薬剤師が入力） |

**補足**: 処方箋の原本は来局時に必要。事前に患者から画像送信があっても、それは「準備の参考」として扱い、原本確認は来局時に行う運用とする。

#### 3.1.1a 患者情報の登録（なるべく簡単に）

| 項目 | 内容 |
|------|------|
| 基本方針 | **入力項目を最小限**にし、登録を簡単にする |
| 必須項目 | 患者名のみ（または名前＋生年月日など最小限） |
| 任意項目 | 電話番号、LINE紐付け、メモなどは後から追加可能 |
| 入力の簡易化 | 処方箋登録時に「既存患者から選択」または「新規（名前だけ入力）」で即登録 |
| よく使う患者 | 検索・選択で素早く選べる。最近使った患者を上に表示するなど |
| 一括登録 | 既存の患者リスト（Excel等）からの取り込みは将来対応として検討 |

#### 3.1.2 ステータス管理

| ステータス | 意味 | 薬剤師の操作 |
|-----------|------|-------------|
| 受付 | 処方箋を受付した | 新規登録時に自動 |
| 準備中 | 調剤・監査中 | 手動で「準備中」に変更 |
| 完了 | 調剤完了・患者に渡せる状態 | 手動で「完了」に変更 |

**完了時の自動通知**: ステータスを「完了」にしたタイミングで、該当患者がLINE友だちであれば、自動で「お薬のご用意ができました。お越しください。」などのメッセージをLINEで送信する。

#### 3.1.3 患者とのチャット

| 項目 | 内容 |
|------|------|
| 目的 | 患者からの質問・相談にLINEで対応する |
| 表示 | 管理画面に「チャット一覧」を表示。患者ごとの会話履歴を見られる |
| 送信 | 管理画面から患者へメッセージを送信（LINE経由で届く） |
| 受信 | 患者がLINEで送ったメッセージが管理画面に表示され、薬剤師に通知 |

**患者からのメッセージ形式**（選択式メイン＋手入力）:
- **選択式をメイン**: 体調・相談内容などをボタンで選んでもらう（タップするだけで送信）
- **手入力も可能**: 選択肢にない内容は、通常のテキスト入力で送信できる

**技術的な仕組み**: 患者がLINEで送信→Webhookでサーバーに届く→DBに保存→管理画面に表示。薬剤師が返信→LINE APIで患者へ送信。

### 3.2 フォローアップ機能

#### 3.2.1 薬渡し時の入力

| 項目 | 内容 |
|------|------|
| タイミング | 患者に薬を渡したとき（または渡す直前） |
| 入力項目 | 患者選択、薬名（複数可）、フォロー日程パターン選択 |
| 入力の簡易化 | よく使う薬・パターンは登録しておき、選択式で入力 |

**例**:
- 患者: 山田太郎（LINE紐付け済み）
- 薬: アモキシシリンカプセル
- パターン: 抗生物質→3日後

→ 3日後に自動でLINE送信が予約される。

#### 3.2.2 フォロー日程パターン

| 項目 | 内容 |
|------|------|
| 登録数 | 上限は固定せず、**増やせる余地を残す**。初期は10パターン程度を目安にし、必要に応じて上限を引き上げ可能な設計 |
| 入力形式 | **フリー入力**。パターン名・送信日数・文例を自分で作成 |
| 編集 | 登録後も**いつでも修正可能** |
| パターン例 | 抗生物質→3日後、解熱鎮痛薬→2日後、胃腸薬→1週間後 など |

**各パターンで登録する項目**:
- パターン名（例: 抗生物質）
- 送信日数（何日後に送るか）
- フォローアップ文例（後述）

#### 3.2.2a フォローアップ文例（メッセージテンプレート）

| 項目 | 内容 |
|------|------|
| 目的 | フォロー送信時のメッセージ文を事前に登録し、使い回す |
| 登録・編集 | 管理画面で**いつでも登録・修正可能** |
| 変数 | `{患者名}` `{薬名}` `{日数}` などを埋め込んで、送信時に自動置換 |
| パターンとの紐付け | 各フォロー日程パターンに1つずつ文例を紐付け |

**文例の例**:
```
{患者名}様、{薬名}をお渡ししてから{日数}日経ちました。体調はいかがですか？
```

**患者への返信選択肢**: 文例に添える「体調選択肢」も**フリー入力で自分で作れる**。デフォルト（とても良い/良い/普通/悪い/とても悪い）のほか、パターンごとにカスタム選択肢を登録可能。

#### 3.2.3 指定日時の自動送信

| 項目 | 内容 |
|------|------|
| 仕組み | 登録した「フォロー予定日時」になったら、LINEでメッセージを自動送信 |
| 送信内容 | 「〇〇様、〇〇（薬名）をお渡ししてから〇日経ちました。体調はいかがですか？」など |
| 実装方法 | サーバー側で定期的に「今日送るべきフォロー」をチェックし、該当があればLINE APIで送信 |

**技術補足**: VercelのCron Jobsや、Supabaseのpg_cronなどで「毎日決まった時間にチェック」する処理を動かす。

#### 3.2.4 患者の返信→薬剤師への通知

| 項目 | 内容 |
|------|------|
| 患者の返信形式 | **選択式をメイン**、**手入力も可能** |
| 選択式 | ボタンタップで送信。**選択肢は自分で作成**（フリー入力） |
| 選択肢の登録 | 管理画面で自由に登録・編集。デフォルト例（とても良い/良い/普通/悪い/とても悪い）も使える |
| 手入力 | 選択肢にない内容は、通常のテキスト入力で送信できる |
| 実装方法 | Quick Reply（ボタン選択）＋通常のテキストメッセージ受信 |
| 薬剤師への通知 | 患者が返信したら、管理画面に「未確認」バッジ表示、またはメール通知（オプション） |

**返信フォーム例**（LINE上で患者が操作。選択肢は登録した内容を表示）:
```
体調はいかがですか？
[登録した選択肢1] [登録した選択肢2] [登録した選択肢3] …

※上記のどれにも当てはまらない場合は、そのままメッセージを入力してください。
```

#### 3.2.5 返信フォームの仕様

| 項目 | 内容 |
|------|------|
| 基本方針 | **選択式をメイン**にし、手入力も可能にする |
| 選択肢 | **フリー入力で自分で作成**。パターンごとにカスタム選択肢を登録可能 |
| 手入力 | 患者が自由にテキストを入力して送信できる（LINEの通常メッセージ） |
| 使い分け | 選択ボタンは「タップするだけ」で楽。詳細や選択肢外の内容は手入力で |
| 送信後のメッセージ | 管理画面で登録・編集可能（例：「ご回答ありがとうございます。気になることがあればお気軽にご連絡ください。」） |

### 3.3 一斉送信機能

**できます。** LINE Messaging API には一斉送信の方法があります。

| 方式 | 送信先 | 用途例 |
|------|--------|--------|
| **ブロードキャスト** | 友だち全員 | 休業のお知らせ、インフルエンザ流行時の注意喚起など |
| **マルチキャスト** | 指定した患者のみ（最大500人/回） | 特定の薬を服用中の患者への一斉フォロー、セグメント配信 |

**管理画面での操作イメージ**:
- メッセージ本文を入力
- 「全員に送信」または「条件で絞って送信」を選択
- 送信前にプレビュー・確認

**料金**: プッシュメッセージと同様、送信人数分が料金対象。例：100人に送信＝100通分。

**実装**: 管理画面に「一斉送信」画面を追加。ブロードキャストAPIまたはマルチキャストAPIを呼び出す。

## 4. 補足・運用前提

### 4.1 LINE公式アカウント

- **既存アカウントを使用**: 新規作成は不要。既存のLINE公式アカウントをMessaging APIと連携させる。
- **友だち追加**: 患者はQRコードをスキャンして友だち追加。薬局のカウンターやチラシにQRコードを掲示。

### 4.2 患者のLINE紐付け

| 方法 | 内容 |
|------|------|
| 友だち追加 | 患者がQRコードで友だち追加 |
| 患者情報の紐付け | 薬局側で「患者名・生年月日」などを管理画面で登録。LINEの友だち追加イベントと、患者IDを紐付ける処理が必要。 |

**紐付けの流れ**（例）:
1. 患者が来局時にQRコードをスキャンして友だち追加
2. 薬剤師が管理画面で「新規患者」を登録し、LINEのユーザーIDと紐付け
3. または、友だち追加時に「患者番号を入力してください」と促し、既存患者と紐付け

**実装の検討**: 友だち追加直後に「患者番号を入力」してもらうか、薬剤師が手動で紐付けるか。運用のしやすさで決める。

### 4.3 処方箋の原本

- 来局時に原本の確認が必須。事前の画像送信は「準備の参考」として扱う。
- システム上は「処方箋受付」と「患者来局・原本確認」を区別するかどうかは、運用で決める。

## 5. 技術要件

### 5.1 使用技術（想定）

**設計方針**: 将来的なマルチテナント（複数薬局）対応を考慮し、全テーブルに `tenant_id` を含める。Phase 1 では単一テナント運用でも、初めからカラムを設けておく。

| 分類 | 技術 |
|------|------|
| フロント | Next.js 14（App Router）、Tailwind CSS |
| バックエンド | Next.js API Routes |
| DB | Supabase（PostgreSQL） |
| 認証 | Supabase Auth（薬剤師ログイン） |
| デプロイ | Vercel |
| LINE連携 | LINE Messaging API |

### 5.2 LINE Messaging API の料金（重要）

| プラン | 月額 | 無料メッセージ数 |
|--------|------|------------------|
| コミュニケーション | 0円 | 200通/月 |
| ライト | 5,000円 | 5,000通/月 |
| スタンダード | 15,000円 | 30,000通/月 |

**料金の対象**:
- **プッシュメッセージ**（薬局から先に送る）: 料金対象
- **応答メッセージ**（患者が先に送ったことへの返信）: 無料

**目安**: 年間処方箋約10,000枚の場合、フォローアップ送信を考慮すると、月数百〜数千通になる可能性あり。**ライトプラン（5,000円/月）**が現実的。

### 5.3 必要なLINE設定

| 設定項目 | 内容 |
|----------|------|
| Messaging API | LINE Developersでプロバイダー・チャネル作成 |
| Webhook URL | VercelにデプロイしたアプリのURLを指定 |
| チャネルアクセストークン | 環境変数に保存（APIキー） |
| チャネルシークレット | Webhook署名検証に使用 |

### 5.4 LINE初回セットアップ手順（詳細）

**ステップ1: LINE Developers にログイン**

1. [LINE Developers](https://developers.line.biz/) にアクセス
2. LINEアカウントでログイン（個人・ビジネスどちらでも可）

**ステップ2: プロバイダー作成**

1. コンソール画面で「Create」→「Create a new provider」
2. プロバイダー名を入力（例: 〇〇薬局）

**ステップ3: Messaging APIチャネル作成**

1. 作成したプロバイダーを選択
2. 「Create a new channel」→「Messaging API」を選択
3. チャネル名（例: 〇〇薬局フォローアップ）
4. **重要**: 「Link to existing LINE Official Account」で、既存の公式アカウントを選択する

**ステップ4: 取得する情報**

| 取得場所 | 項目 | 用途 |
|----------|------|------|
| Messaging API タブ | Channel access token | メッセージ送信に必要 |
| Basic settings タブ | Channel secret | Webhook署名検証に必要 |

**ステップ5: Webhook設定（開発後）**

1. Messaging API タブの「Webhook URL」に、`https://あなたのアプリ.vercel.app/api/line/webhook` を入力
2. 「Use webhook」をONにする
3. 「Verify」で接続確認

**注意**: Webhookはアプリをデプロイしてから設定する。開発中はngrokなどでローカルに転送することも可能。

## 6. データベース設計（概要）

**マルチテナント対応の考慮**: 将来的な複数薬局（テナント）対応を見据え、**全テーブルに `tenant_id` を含める**。Phase 1 では単一テナント想定でも、初めからカラムを設けておく。

| テーブル | 主な項目 |
|----------|----------|
| tenants | テナントID、薬局名、LINEチャネル設定など（マルチテナント時に使用） |
| prescriptions | tenant_id、処方箋ID、患者ID、受付日時、ステータス、処方元、薬名など |
| patients | tenant_id、患者ID、名前、生年月日（任意）、電話番号（任意）、LINE user ID、紐付け日時 |
| follow_up_patterns | tenant_id、パターン名、送信日数、文例テンプレート、返信選択肢（JSON等）、送信後メッセージ（※パターン数は増やせる設計） |
| follow_up_schedules | tenant_id、患者ID、薬名、パターンID、送信予定日時、送信済みフラグ |
| follow_up_replies | tenant_id、患者の返信内容、返信日時、確認済みフラグ |
| chat_messages | tenant_id、患者ID、送信者（患者/薬剤師）、内容、送信日時 |

※ `tenant_id` はテナント（薬局）を識別するID。`tenants` テーブルで薬局情報を管理し、RLS（Row Level Security）でテナントごとにデータを分離する。

**フォローアップ文例・選択肢**: `follow_up_patterns` に文例テンプレート（`{患者名}` 等の変数対応）と返信選択肢（フリー入力で登録）を格納。いずれも管理画面から編集可能。**パターン数の上限はハードコードせず**、設定テーブルや環境変数で管理し、将来の増加に対応できるようにする。

## 7. 実装フェーズ（推奨順序）

### Phase 1: 基盤構築

1. Next.js + Supabase + Vercel のプロジェクト作成
2. 薬剤師ログイン（Supabase Auth）
3. 患者テーブル・処方箋テーブルの作成
4. 患者登録（最小限の入力で簡単に、既存患者から選択も可能）
5. 処方箋一覧画面（CRUD）

### Phase 2: LINE連携

1. LINE Developersでチャネル作成・Messaging API有効化
2. Webhook受信の実装（友だち追加・メッセージ受信）
3. 患者とLINE user IDの紐付けフロー
4. プッシュメッセージ送信（完了通知）

### Phase 3: フォローアップ

1. フォロー日程パターンの登録機能（パターン数は増やせる設計、文例・選択肢はフリー入力で編集可能）
2. 薬渡し時の入力画面
3. スケジュール送信（Cron等）
4. 患者返信フォーム（Quick Reply）
5. 薬剤師への通知表示

### Phase 4: チャット

1. チャット履歴表示
2. 薬剤師からの送信
3. 患者メッセージの管理画面表示

## 9. よくある疑問・注意点

### Q1: 既存のLINE公式アカウントを連携すると、今の使い方は変わる？

**A**: Messaging APIを有効にすると、**応答メッセージ**（自動返信）の設定が無効になります。代わりに、Webアプリ側で受信したメッセージを処理できます。手動で返信する運用はこれまで通り可能です。

### Q2: 患者がLINEをブロックしたら？

**A**: プッシュメッセージは届きません。料金もカウントされません。管理画面では「送信失敗」として記録するか、次回来局時に再度友だち追加を案内する運用を検討してください。

### Q3: 個人情報は大丈夫？

**A**: 患者名・薬名・体調情報は医療関連の個人情報です。SupabaseのRLS（Row Level Security）でアクセス制限し、Vercelの環境変数でAPIキーを管理します。LINEのデータはLINEの利用規約に従います。

### Q4: 処方箋の画像を送ってもらう機能は作る？

**A**: 要件では「事前送信は準備用途」としています。画像受信機能を入れると、保存・管理・セキュリティの負荷が増えます。Phase 1では手動登録のみとし、必要に応じて後から追加するのがおすすめです。

### Q5: 複数薬剤師で使う場合

**A**: Supabase Authで複数アカウントを登録し、誰がどの処方箋を担当したか記録する項目を追加できます。Phase 1では単一ユーザー想定で進め、必要になったら拡張します。

## 10. 用語集（非エンジニア向け）

| 用語 | 説明 |
|------|------|
| API | システム同士がデータをやり取りするための「窓口」。LINE APIなら、あなたのアプリからLINEにメッセージを送れる |
| Webhook | 何かが起きたときに、自動で相手のサーバーに通知する仕組み |
| 環境変数 | パスワードやAPIキーなどの機密情報を、コードに直接書かずに管理する仕組み |
| Cron | 決まった時間に自動で処理を実行する仕組み |
| Supabase | データベースと認証を一括で提供するサービス |
| Vercel | Webアプリをインターネットに公開するサービス |
